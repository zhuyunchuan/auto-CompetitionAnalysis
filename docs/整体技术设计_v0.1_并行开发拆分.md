# 整体技术设计 v0.1 - 并行开发拆分（OpenClaw 竞品抓取）

## 1. 目标与原则
1. 目标：在云端完成双品牌（Hikvision/Dahua）动态层级抓取、参数抽取、标准化、异常检测和 Excel 导出。
2. 并行原则：按“低耦合、高内聚”拆分，子模块写入路径互不重叠。
3. 当前默认存储：`SQLite + Parquet`，后续可迁移 `PostgreSQL`。

## 2. 系统分层
1. 编排层：OpenClaw DAG 与任务调度。
2. 采集层：品牌适配器、层级发现、目录抓取、详情抓取。
3. 解析层：字段抽取、标准化、规则引擎。
4. 数据层：SQLite 持久化、Parquet 批次快照。
5. 交付层：Excel 导出、运行摘要、异常输出。

## 4. 核心接口契约
## 4.1 统一数据对象（`src/core/types.py`）
1. `HierarchyNode`
   - `brand, series_l1, series_l2, source, status, discovered_at`
2. `CatalogItem`
   - `brand, series_l1, series_l2, model, name, url, locale`
3. `SpecRecord`
   - `run_id, brand, series_l1, series_l2, model, field_code, raw_value, normalized_value, unit, source_url, confidence`
4. `QualityIssue`
   - `run_id, brand, series_l1, series_l2, model, issue_type, field_code, detail, severity`

## 5. OpenClaw DAG 设计
1. `discover_hierarchy`
   - 输入：品牌入口 URL
   - 输出：`hierarchy_snapshot`
2. `crawl_product_catalog`
   - 输入：层级快照
   - 输出：`product_catalog`
3. `fetch_product_detail`
   - 输入：产品目录
   - 输出：原始 HTML 快照
4. `extract_and_normalize_specs`
   - 输入：HTML + 目录上下文
   - 输出：`product_specs_long`
5. `detect_data_quality_issues`
   - 输入：`product_specs_long`
   - 输出：`data_quality_issues`
6. `merge_manual_inputs`
   - 输入：manual_append
   - 输出：覆盖后 specs
7. `export_excel_report`
   - 输入：catalog/specs/issues/run_summary
   - 输出：`competitor_specs_<run_id>.xlsx`

## 6. 并行开发拆分（子 Agent 分工）
以下分工按“文件写入域不重叠”设计，可并行编码。

1. Agent-A（Core/Config）
- 负责：`src/core/*`
- 输出：配置加载、类型定义、通用常量、日志初始化
- 不改动：`adapters/`, `extractor/`, `pipeline/`

2. Agent-B（Storage）
- 负责：`src/storage/*`
- 输出：SQLite schema、Repo 层、Parquet 存储
- 不改动：`adapters/`, `pipeline/`, `export/`

3. Agent-C（Hikvision Adapter）
- 负责：`src/adapters/hikvision_adapter.py`
- 输出：Hikvision 层级发现、目录抓取、详情拉取
- 不改动：`dahua_adapter.py`, `extractor/*`

4. Agent-D（Dahua Adapter）
- 负责：`src/adapters/dahua_adapter.py`
- 输出：Dahua 层级发现、目录抓取、详情拉取
- 不改动：`hikvision_adapter.py`, `extractor/*`

5. Agent-E（Extractor/Normalizer）
- 负责：`src/extractor/*`
- 输出：字段抽取器、单位归一、分辨率/FPS 解析
- 不改动：`adapters/*`, `pipeline/*`

6. Agent-F（Quality Rules）
- 负责：`src/quality/*`
- 输出：缺失字段、解析失败、单位异常等规则
- 不改动：`extractor/*`, `pipeline/*`

7. Agent-G（Pipeline/DAG）
- 负责：`src/pipeline/*`
- 输出：任务编排、节点依赖、重试与告警
- 不改动：`adapters/*`, `quality/*`, `extractor/*`

8. Agent-H（Export/Manual）
- 负责：`src/export/*`, `src/manual/*`
- 输出：Excel 导出、人工补录导入与覆盖逻辑
- 不改动：`pipeline/*`, `adapters/*`

## 7. 依赖关系与集成顺序
1. 先完成：Agent-A + Agent-B（基础设施）
2. 并行进行：Agent-C、Agent-D、Agent-E、Agent-F
3. 后置集成：Agent-G（串联全链路）
4. 最后接入：Agent-H（Excel 与人工回写）

## 8. 联调契约（必须冻结）
1. `field_code` 字典冻结：`docs/field_dictionary_v1.md`
2. SQLite 表结构冻结：`src/storage/schema.py`
3. Adapter 返回结构冻结：`CatalogItem` / `SpecRecord`
4. Excel 列顺序冻结：`export/excel_writer.py` 常量定义

## 9. Definition of Done（每个 Agent）
1. 单元测试覆盖核心逻辑（至少成功路径+1个异常路径）。
2. 不引入跨模块循环依赖。
3. 提交前通过 `ruff + pytest`。
4. 输出变更清单：文件路径 + 接口变化点。

## 10. 集成风险与规避
1. 风险：字段命名不一致导致拼接失败
- 规避：统一 `types.py` + schema 常量。

2. 风险：两个品牌页面变化导致选择器失效
- 规避：适配器层独立回归样例、失败降级到告警。

3. 风险：并行开发冲突
- 规避：严格按模块写入域分工，不跨目录改动。
